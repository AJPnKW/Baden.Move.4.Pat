# Issue Log

| Issue | Impact | Response |
| --- | --- | --- |
| Scope was not fully known at the start | Early planning could not be fully locked down | Used planning-to-plan and progressive clarification |
| Renovation choices affected occupancy readiness | Delays or wrong sequencing could cascade | Treated renovation work as dependency-driven planning |
| Move activities had multiple transport paths | Packing and timing became more complex | Split work by mover, self-move, and disposition stream |
| Time-sensitive cutover tasks clustered around possession | Risk of missed tasks increased | Used day-specific readiness and cutover framing |
| Personal project context created frequent change pressure | Rework and drift risk increased | Captured changes and decisions at a high level |
| Research and analysis produced new requirements | Earlier assumptions could become obsolete | Reframed learning work as requirement and feasibility validation |
| Some workstreams looked small but had broad downstream impact | Underestimating them could destabilize other work | Escalated those items into phase-level planning rather than treating them as minor tasks |
| Private household information existed beside legitimate project artifacts | Public reporting risked oversharing | Split public case-study reporting from private workspace detail |

## Issue Handling Pattern

Issues were not handled as a formal enterprise risk register. Instead, the project used a lightweight pattern:

1. identify what changed or what became risky
2. decide whether the issue was local or system-wide
3. handle local issues with agile adjustment
4. handle system-wide issues with tighter sequencing and broader change control
